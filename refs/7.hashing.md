# Part I  
## Hashing

---

# General Idea

A **hash table** is an **array of fixed size** containing data items.

Properties:

- Each item has a **key**
- Search is performed based on the **key**
- Keys are mapped to array positions

Mapping rule:

```
0 ... m-1
```

where:

```
m = table size
```

This mapping is done by a **hash function**.

Example applications:

- **Compilers**: symbol tables
- **Spell checkers**
- **Database indexing**

---

# Hash Table

A hash table supports:

- search
- insertion
- deletion

Average complexity:

```
O(1)
```

Characteristics:

- Data is **not ordered**
- Cannot efficiently support:

```
find min
find max
successor
predecessor
range queries
sorted listing
```

---

# Unrealistic Solution: One Slot per Key

Universe of keys:

```
U = all possible keys
```

Hash table:

```
T[k]
```

Meaning:

```
T[k] stores the data item with key k
```

If no item exists:

```
T[k] = NIL
```

Operations:

```
search = O(1)
insert = O(1)
delete = O(1)
```

Problem:

If key universe is large, **memory waste is huge**.

Example:

```
HKUST student ID
8 digits → 10^8 possible keys
but only ~7000 students
```

---

# Hash Function

A hash function:

```
h : U → {0,1,...,m-1}
```

Maps keys into hash table slots.

Example:

```
h(k) = slot index
```

Multiple keys may map to the **same slot**.

---

# Hash Function Mapping

```
possible keys → hash function → hash table
{k0,k1,...,k|U|-1} → {0,1,...,m-1}
```

Typically:

```
m << |U|
```

Hash value:

```
h(k)
```

Storage:

```
item with key k stored at T[h(k)]
```

---

# Collision Problem

Collision:

```
two keys → same hash slot
```

Impossible to avoid if:

```
N > m
```

where

```
N = number of keys
m = table size
```

Solutions:

1. **Design good hash function**
2. **Use collision resolution strategy**

---

# Hash Function Design

Common simple function:

```
h(k) = k mod m
```

Example:

```
m = 12
k = 100
h(k) = 4
```

Advantages:

- very fast

Important considerations:

- avoid powers of 2
- avoid powers of 10
- choose **prime table size**

Example:

```
expected 2000 items
m ≈ 701 (prime)
```

---

# String Key Hashing: Method 1

Convert string to number:

```
sum ASCII values
```

Example idea:

```
key = Σ character values
```

Problems:

- permutations collide

Example:

```
abc
cab
bca
```

same hash value.

Distribution also poor for large tables.

---

# String Key Hashing: Method 2

Polynomial hash:

```
h(key) = (key[0] + 27*key[1] + 27²*key[2]) mod m
```

Better distribution.

Problems:

- first characters not random
- English words limited combinations

Example:

```
only ~2851 possible first 3-letter combinations
```

---

# String Key Hashing: Method 3

Better polynomial hashing:

```
h(key) = Σ 37^(L-1-i) * key[i] mod m
```

Where:

```
L = key length
```

Advantages:

- uses **all characters**
- better distribution

# Part II  
## Collision Handling

---

# Method 1: Separate Chaining

Idea:

```
each slot stores a linked list
```

Keys with same hash value form a **chain**.

Example:

```
hash(k) = k mod 10
```

Table entry:

```
T[i] → linked list of keys
```

---

# Separate Chaining: Insertion

Algorithm:

1. compute

```
h(k)
```

2. check table slot

```
T[h(k)]
```

3. if empty:

```
create list
```

4. otherwise:

```
insert at head of list
```

---

# Separate Chaining: Deletion

Steps:

1. compute

```
h(k)
```

2. traverse list

3. remove node containing key.

---

# Separate Chaining Characteristics

If hash function distributes well:

```
average list size = small constant
```

Complexity:

```
search = O(1)
insert = O(1)
delete = O(1)
```

Advantages:

- deletion simple

Disadvantages:

- linked list memory overhead
- slower due to allocation.

---

# Method 2: Open Addressing

Instead of linked lists:

```
all elements stored in table
```

If collision occurs:

```
relocate element
```

Important questions:

- relocation scheme
- search procedure

---

# Open Addressing Strategies

Three common methods:

```
1 linear probing
2 quadratic probing
3 double hashing
```

---

# Basic Open Addressing Formula

Probe sequence:

```
hi(k) = (hash(k) + f(i)) mod m
```

Where:

```
i = probe number
f(0) = 0
```

---

# Linear Probing

Function:

```
f(i) = i
```

Probe formula:

```
hi(k) = (hash(k) + i) mod m
```

Algorithm:

```
for i = 0 to m-1
    j = (hash(k) + i) mod m
    if T[j] empty
        insert
```

---

# Linear Probing Example

Hash function:

```
hash(k) = k mod 10
```

Insert keys:

```
89, 18, 49, 58, 69
```

Probe example:

```
58 → T[8],T[9],T[0],T[1]
69 → T[9],T[0],T[1],T[2]
```

---

# Primary Clustering

Cluster:

```
contiguous block of filled cells
```

Problem:

- clusters grow over time
- insert time increases

Effects:

- clusters merge
- performance degrades quickly

---

# Quadratic Probing

Function:

```
f(i) = i²
```

Probe formula:

```
hi(k) = (hash(k) + i²) mod m
```

Example:

```
hash(k) = k mod 10
```

Probe sequence example:

```
58 → T[8],T[9],T[(8+4) mod 10]
```

---

# Quadratic Probing Properties

Advantages:

- reduces **primary clustering**

Condition:

If table size **m is prime**:

```
insertion possible if table ≤ 50% full
```

---

# Secondary Clustering

Occurs when:

```
two keys have same initial hash
```

They follow identical probe sequence.

Impact:

- slower searches.

Solution:

```
make probe sequence depend on key
```

---

# Double Hashing

Use two hash functions.

Probe formula:

```
hi(k) = (hash(k) + i*hash2(k)) mod m
```

Example:

```
hash2(k) = R - (k mod R)
```

where

```
R < m
R is prime
```

---

# Double Hashing Example

Parameters:

```
m = 10
R = 7
hash(k) = k mod 10
hash2(k) = 7 - (k mod 7)
```

Insert:

```
89, 18, 49, 58, 69
```

Probe example:

```
49 → T[(9 + 7) mod 10]
58 → T[(8 + 5) mod 10]
69 → T[(9 + 1) mod 10]
```

---

# Choosing hash2()

Requirements:

```
hash2(k) ≠ 0
```

Also:

```
gcd(hash2(k), m) = 1
```

Otherwise:

only part of table will be probed.

Example bad case:

```
hash(k) = 0
hash2(k) = m/2
```

Only two slots visited.

Recommended setup:

```
m prime
R prime < m
hash2(k) = R - (k mod R)
```

# Deletion in Open Addressing

Direct deletion breaks probe sequence.

Solution:

```
lazy deletion
```

Each slot has state:

```
EMPTY
ACTIVE
DELETED
```

---

# Rehashing

Load factor:

```
α = N / m
```

where

```
N = number of stored items
m = table size
```

If α becomes large:

```
performance degrades
```

Solution:

```
increase table size
rehash all elements
```

Steps:

1. create larger table
2. recompute hash for each key
3. insert into new table

Rehashing cost is **high**, but occurs **rarely** in good implementations.
